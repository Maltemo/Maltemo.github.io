Where it all began

I did participate in December 2023 in a Capture The Flag event called PingCTF.

I wrote a detailed explanation about the challenge at the time on HackMD website.

HackMD is an online collaborative web editor for markdown that I used a lot during my studies. I still use it to get nice responsive articles in my website.

In January 2024, I presented this topic as a short presentation at Drink Love Share V1, an event hosted by Laluka. During this presentation, I spoke in french. This presentation is available on youtube.

At the end of this presentation, I promised I would upload the full write-up in english the next day.

Later that night, I connected on HackMD and tried to export my markdown write-up in an HTML document, in order to upload it then on my personal website.

But the exported document was completly messed up, as if CSS document wasn’t loaded or if the HTML tags were scrambled.

I started investigating in the JavaScript console to check if a script wasn’t correctly loaded. But it didn’t give me any hints.
I disabled any extensions potentialy blocking JS, but it didn’t change the state of the exported report.

Then I inspected HTML and bingo : I found that the value $' was replaced.

As it was late and I had identified the source of the problem, I just decided to repare the report by escaping $ characters with $$ (see Mozilla documentation on the matter).

Understanding the root cause

The payload inside my write-up was triggered during the export process from markdown to HTML

In JavaScript, replace() and replaceAll() functions accepts specific characters that will be interpreted differently

Pattern Inserts
$$ Inserts a "$".
$& Inserts the matched substring.
$` Inserts the portion of the string that precedes the matched substring.
$' Inserts the portion of the string that follows the matched substring.
$n Inserts the nth (1-indexed) capturing group where n is a positive integer less than 100.
$<Name> Inserts the named capturing group where Name is the group name.

Copy of the table from Mozilla documentation

$ is the key character enabling to insert data when it appears in the replacement argument of a replace function.

I already explained in detail how $' operator works in this section of the same write-up that made me discover the following vulnerability.

If you don’t have time to read it, $' is a placeholder in the value replaced for : “take everything which is after the token replaced”.

Going further

By default, HackMD sanitizes the HTML or Markdown during the export process.

As the original payload was only breaking the page, I was curious if I was able to get an XSS and pop an alert window !

The HTML isn’t correctly sanitized because the value that is cleaned up still contains the $' character, before it’s replaced.

After replacement, this breaks the DOM that was sanitize and allow to create our XSS. This trick is explained in detail in my previous article.

I created a private note, to avoid impacting other users and tried to reproduce the bug based on the report.

I quickly identified were the problem came from.

The payload had to be in an h1,h2, etc. tag, and had to be inside a code section (backticks in markdown).

Here was a valid XSS payload:

### Test `$' "<img src=x onerror=alert(1)>` XSS

The payload was triggered 26 times.

This vulnerability wont work anymore as it has already been patched after I reported it.

Based on the structure of the exported HTML document, the vulnerability came from the generation of the "Table of Content" also called TOC.

My guess is that the TOC on the right side of the page was generated during the export with a JavaScript replace function.

I didn’t have access to the source code of HackMD as this service is a closed source one.

This same service is available as CodiMD, which is the open-source alternative maintained by the community and the same company.

I wasn’t able to reproduce this vulnerability on CodiMD with the latest version, the export probably didn’t have the same code.

I was sad because I really wanted to provide a full patch for the vulnerability and dive in the details and reasons why this vulnerability was exploitable, but unfortunately HackMD is a closed source software.

Vulnerability impact

The XSS is stored inside the exported HTML.

A local XSS can’t access local files of the victim.
It can’t steal cookies/local storage of other pages in the browser.
But it can redirect to a specific page, modify completely the current page, simulate a login page, etc.

An attacker could have scripted the addition of the payload inside every public editable shared notes of HackMD and wait for users to export the HTML report, access it and pop the XSS payload.

The next step could have been a redirection to a phishing website looking like HackMD (he can just copy CodiMD) and retrieve every trapped users’ credentials.

Reporting timeline

  • 16th february 2024 - I contacted HackMD through their discord and got in touch with on of their developer the .
  • 19th february 2024 - I sent the vulnerability details in an encrypted email to support@hackmd.io
  • 7th march 2024 - The developer contacted me again to check if the vulnerability was corrected in production. Also got the authorization to publish an article about my findings.
  • 14th march 2024 - Security fix mentionned in their release note document : https://hackmd.io/s/release-notes#-EE-1682-2024-03-06-1631

Conclusion

HackMD did mention this vulnerability in their patch note without mentioning me but honestly I don’t care.

The only thing that matters is that the vulnerability was corrected and that I helped in the process.

I often hear : "CTFs have nothing to do with the real world"

Guess they were wrong
